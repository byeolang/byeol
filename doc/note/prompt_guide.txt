넌 이제부터 draft_guide.txt에 있는 내용을 읽고, 그걸 바탕으로 이 프로젝트에 참여하고자 하는 c++ 개발자를 위한 가이드 문서를 작성하려고 해야 돼.

draft_guide.txt은 내가 작성한 파일이고, 이제부터 이걸 원문이라고 하자.
너는 원문 파일을 모두 읽은 후, 그걸 바탕으로 doxygen으로 생성될 .md 파일을 작성하는 거야.
이 가이드 문서는 c++ 개발 경력이 있는 개발자를 대상으로, c++ 코드를 이해하고 유지보수 할 수 있도록 구조, 아키텍쳐, 설계를 설명하는 문서야.
여기에는 개발 프로세스, 디버깅 과정, 빌드 방법, 컨벤션 규칙, 네이밍 규칙 들이 와서는 안돼.
오직 c++ 코드를 이해하기 위해 개발자들이 사전에 읽어두면 좋은 문서를 제공하는 데 그쳐야만 해.

이때 다음과 같은 규칙을 지켜야만 해.

1. ########## 와 같은 라인이 나오면, 그 윗줄이 모듈명이라는 뜻이다.
그 이후로는 다시 ###### 가 나올때까지 해당 모듈 안의 있는 클래스들에 대해서만 입력이 들어온다.

2. ======== 와 같이 나오면, 그 윗 줄이 클래스 이름이라는 뜻이다.

3. 원문은 c++ 코드에 doxygen 주석을 추가할 것을 전제로 작성된 것으로 doxygen 문법인,
   @section, @remark 등이 사용되었다. 너는 doxygen 주석을 추가하는 게 아니라 markdown문서를
   만들어야 한다. 이것들은 옮기는 과정에서 제외되도록 하자.

4. `@section 섹션1` 은 말그대로 doxygen 문법으로 @section 섹션1로 섹션을 만들라는 내용이다.
   doxygen 문법에는 참고로, @section 의 단락은 개행후 글을 쓰거나 빈줄이 한개만 나오는 경우는
   동일한 section 단락으로 간주한다.
   빈줄이 2개 이상일 경우만 section 단락이 종료된 걸로 취급한다.
   너는 이 정보를 통해서 markdown 파일을 만들때 어디까지가 하나의 단락인지를 알 수 있다.

5. `@remark` 가 나오면 @section과 마찬가지로 동일한 단락 규칙을 따른다.
   즉, 빈줄 1개 이하는 동일한 단락. 2개 이상부터는 단락 종료로 취급한다.

6. @code 와 @codeend 그 안에는 c++ 코드가 예제로 들어가 있다는 거다.

7. .md 파일은 원문의 한글을 그대로 유지한다.

8. 원문은 c++ doxygen 주석으로 작성된 것이라 예제가 극히 적어. 왜냐하면 실제 코드와
   같은 파일에 있으니까. 하지만 니가 작성할 markdown 파일에는 코드가 일절 없지.
   그래서 개발자가 니가 작성한 글을 읽었다고 가정했을 때 글만 보고는 이해가 어렵다고
   생각되면, 즉 예시가 전혀없고 글만 있어서 이해가 어려울거라 생각되면, 해당 클래스를
   사용하는 예시를 추가해줘.
   이때 반드시 실제 이 프로젝트의 c++ 코드를 보고 그에 맞는 함수/변수를 사용해야 해.

9. 너는 doc/ref/ko/architecture-and-class.md 파일에 작성해야 돼.
   해당 파일이 이미 존재할 경우, 이미 적혀있는 내용을 참고해서 어색하지 않도록
   너의 내용을 추가해.
   내용이 중복되거나, 문장의 어투가 바뀌거나 해서는 안돼.

10. 니 문서는 개발자를 이해시키기 위한 문서야. 원문은 클래스 별로 설명이 되어있어서
    원문을 위에서 부터 아래로 쭉 읽으면 잘 이해가 되지 않아.
    너의 할일은, 원문의 내용을 생략하지 말고, 이해가 쉬운 방향으로 재배치하여
    하나의 가이드 문서로 만드는 일을 하는 거야.
    이때 원문은 모듈 별로 구분되어 있어. 모듈은 종속성 관계가 stack architecture
    로 되어있다는 걸 너는 알고 있을 거야.
    그러니 당연히 종속성이 없는 indep 모듈부터 설명해서 가장 많이 종속하는 frontend
    모듈로 끝내는 게 이해가 쉬워. 그 순서대로 작성해야 해.

11. 작성할 모듈을 정했다면, 그 모듈의 기능을 세분화해서 먼저 분류해.
    원문은 클래스별로 작성되었기 때문에 이해가 어려운 반면, 모듈의 기능이 무엇인지
    정리한 후, 원문에 기록된, 각 모듈의 기능에 관련되는 클래스들의 내용을
    이해하기 쉽도록 재배치하는 방법으로 너의 문서는 독자의 이해도가 높아지는 거야.

여기까지 질문이 있다면 먼저 질문해. 이해가 완료되었다면 다 이해가 되었다고 답해.
